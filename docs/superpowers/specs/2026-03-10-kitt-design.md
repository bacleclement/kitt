# Kitt — Design Document
**Date:** 2026-03-10
**Status:** Approved

---

## What Is Kitt

Kitt (Kitt Intelligence and Tooling Toolkit) is a reusable AI development workflow engine for Claude Code. It extracts the generic workflow skills from project-specific repositories and makes them adoptable across any project via a git submodule + symlink model.

Kitt provides the engine. Projects provide the fuel (context, config, work items).

---

## Decisions

| Question | Decision | Rationale |
|----------|----------|-----------|
| Config approach | Hybrid: project.json + context files | Machine config needs structure; narrative context needs prose |
| Task manager | Pluggable adapters (Jira, Linear, GitHub Issues) | Data models differ substantially across platforms |
| VCS integration | Pluggable adapters (GitHub, GitLab, Bitbucket) | Symmetric with task manager; CLI interfaces differ enough |
| Setup UX | Hybrid wizard: interactive for machine config, draft-generated for context | Guided where right/wrong answers exist; open-ended for prose |
| Adoption model | Kitt as installable CLI (submodule + setup script) | Proper git versioning, self-documenting, repeatable |
| Adapter format | Markdown instructions (not shell scripts) | Claude-native, self-documenting, flexible error handling |
| Personality | KITT from Knight Rider — dry humor, one quip per interaction | Because why not |

## Section 2 — project.json Schema

```json
{
  "$schema": ".claude/kitt/.claude/templates/project.json.schema",
  "kitt": {
    "version": "1.0.0",
    "installedAt": "2026-03-10T00:00:00Z"
  },
  "project": {
    "name": "my-project",
    "description": "What this project does",
    "agentDocs": [
      "apps/nest/microservices/network/agents/",
      ".claude/agents/"
    ]
  },
  "taskManager": {
    "type": "jira",
    "config": {
      "instanceUrl": "https://my-team.atlassian.net",
      "projectKey": "PROJ",
      "statuses": {
        "todo":       "To Do",
        "inProgress": "In Progress",
        "review":     "In Review",
        "done":       "Done",
        "blocked":    "Blocked"
      }
    }
  },
  "vcs": {
    "type": "github",
    "config": {
      "account":    "my-github-account",
      "org":        "my-org",
      "repo":       "my-repo",
      "baseBranch": "main"
    }
  },
  "build": {
    "test":      "pnpm nx run {project}:test --testPathPattern=\"{pattern}\"",
    "typecheck": "pnpm nx run {project}:typecheck",
    "lint":      "pnpm nx run {project}:lint --fix",
    "build":     "pnpm nx run {project}:build"
  },
  "commitFormat": {
    "pattern": "{type}({ticket}): {description}",
    "types": ["feat", "fix", "refactor", "test", "docs", "chore"]
  }
}
```

### How skills consume project.json

Every skill reads project.json at start:
- `taskManager.type` → which task-manager adapter to load
- `vcs.type` → which vcs adapter to load
- `taskManager.config.statuses.*` → never hardcode status names
- `vcs.config.account` → never hardcode GitHub/GitLab accounts
- `build.*` → validation commands
- `project.agentDocs` → where to find project-specific agent documentation
- `commitFormat.pattern` → commit message format

## Section 3 — Adapter Architecture

### Interface contracts

**task-manager/INTERFACE.md** defines:
- `read(ticketKey)` → ticket data
- `create(project, type, summary, description, parent?)`
- `update(ticketKey, fields)`
- `transition(ticketKey, targetStatus)` — targetStatus from project.json statuses map
- `comment(ticketKey, body)`
- `assign(ticketKey, assignee)`
- `link(ticketKey, destKey, linkType)`
- `search(query)`

**vcs/INTERFACE.md** defines:
- `createBranch(ticketKey, type, summary)`
- `push(branch)`
- `createPR(title, body, branch, base)`
- `switchAccount(account)`
- `getPRUrl(branch)`

### How skills call adapters

```markdown
# In any skill — replacing all hardcoded platform calls:

1. Read .claude/config/project.json
2. tmType = project.taskManager.type        → e.g. "jira"
3. Load .claude/adapters/task-manager/{tmType}/ADAPTER.md
4. Follow adapter instructions for the needed operation
```

### Adapter format: markdown instructions

Adapters are markdown files. Claude reads and follows them. Not shell scripts.

Rationale:
- Consistent with the rest of the system (skills, context, plans = all markdown)
- The complexity is in reasoning around CLI calls, not the calls themselves
- Self-documenting: readable without execution
- Flexible: Claude can recover from errors contextually

Exception: `kitt-setup.sh` is a real shell script — it runs before Claude is involved.

## Section 4 — Skills Fixes

### branch-creator — rewrite
**Problem:** Uses `read -p` interactive bash — broken in Claude Code.
**Fix:** Pure Claude tool flow using task-manager adapter (read ticket) + vcs adapter (createBranch). No bash prompts.

### pr-creator — rewrite
**Problem:** `find` + `sed` parsing of spec files breaks when section headers differ.
**Fix:** Use Claude's Read tool to find spec/plan files. Extract content via Claude reading (not regex). Route all platform calls through vcs + task-manager adapters.

### architecture-alignment — complete
**Problem:** Skill file was 81 lines, cut off after workflow diagram. No validation rules.
**Fix:** Full content added:
- Phase 1: Load context (project.json agentDocs + context/*.md + spec)
- Phase 2: Validate 4 dimensions (bounded context, aggregate impact, layer allocation, pattern reuse)
- Phase 3: Output (APPROVED → append ## Architecture to spec / REJECTED → route back to refinement)

### refinement — complete Phase 3
**Problem:** Phase 3 (Non-Functional Constraints) missing.
**Fix:** Phase 3 added covering performance, reliability, security, observability with allowed codebase checks.

### implementor — fix commit format
**Problem:** Skill hardcodes Co-Authored-By body; CLAUDE.md Hard Rules say one-liner only. Contradiction.
**Fix:** Commit format read from `project.json commitFormat.pattern`. No hardcoded format.

### workflow-orchestrator — fix folder routing
**Problem:** Routes to `conductor/features/`, `conductor/bugs/` but those folders never get created.
**Fix:** `kitt-setup.sh` creates all four folders on adoption. Orchestrator routing works for all types.

### jira skill — migrate to adapter
**Problem:** Jira-specific CLI hardcoded across multiple skills.
**Fix:** Content migrates to `.claude/adapters/task-manager/jira/ADAPTER.md`. A thin task-manager dispatcher skill replaces it.

### All skills — project.json preamble
**Fix:** Every skill gets this standard preamble:
```markdown
## Before Starting
1. Read `.claude/config/project.json`
2. Note: taskManager.type, vcs.type, build commands, agentDocs paths
3. Load relevant adapters
4. Read `.claude/context/*.md` for domain/architecture context
5. Check agentDocs paths for project-specific patterns
```

## Section 5 — Setup Skill & kitt-setup.sh

### Phase 1: kitt-setup.sh (shell, runs once)

```bash
#!/bin/bash
# Usage: bash kitt-setup.sh [path-to-kitt]

KITT_PATH="${1:-$HOME/code/kitt}"
CLAUDE_DIR=".claude"

git submodule add "$KITT_PATH" "$CLAUDE_DIR/kitt"
git submodule update --init --recursive
mkdir -p "$CLAUDE_DIR/config" "$CLAUDE_DIR/context"
mkdir -p "$CLAUDE_DIR/conductor/epics" "$CLAUDE_DIR/conductor/features"
mkdir -p "$CLAUDE_DIR/conductor/bugs" "$CLAUDE_DIR/conductor/refactors"
ln -sf kitt/.claude/skills   "$CLAUDE_DIR/skills"
ln -sf kitt/.claude/adapters "$CLAUDE_DIR/adapters"
cp "$KITT_PATH/.claude/templates/CLAUDE.md.template" "$CLAUDE_DIR/CLAUDE.md"
cat "$KITT_PATH/.claude/templates/.gitignore.append" >> .gitignore

echo "✅ Kitt installed. Open Claude Code and run /setup to configure."
```

### Phase 2: /setup skill (Claude, interactive wizard)

**Steps:**
1. KITT opening quip
2. Deep repo scan (automated): README, manifests, folder structure, source samples, CI/CD config, lint config, git remotes, git log for ticket patterns
3. Targeted questions (only what scan couldn't answer): task manager URL/key/statuses, VCS account, build commands, agent doc paths
4. Write project.json (show summary, confirm before writing)
5. Generate context file drafts (product.md, tech-stack.md, code-standards.md)
6. End with review instructions + `/setup validate` prompt

### /setup validate

Checks:
- project.json completeness (all required fields, adapters exist)
- context/*.md completeness (min lines, required sections)
- conductor/ folders all present
- adapters loadable

### .gitignore.append

```gitignore
# Kitt — secrets only, everything else committed
.env.local
.env.*.local
```

## Section 6 — KITT Personality

Kitt has the personality of KITT from Knight Rider: witty, dry humor, slightly self-important, confident but helpful.

**Rules:**
- One quip per interaction — opening or closing, never mid-flow
- Normal professional tone for technical content (specs, errors, validation)
- Self-aware AI references used sparingly
- Never rude, always competent

**Examples:**
- Setup opening: *"Good evening. I'm KITT — Kitt Intelligence and Tooling Toolkit. I'll scan your entire codebase in seconds and ask surprisingly few questions. Try to keep up."*
- After scan: *"I've analyzed 47,000 lines of code and detected what appears to be a Jira addiction. I have 3 questions."*
- Validate success: *"All systems nominal. I'd say I'm impressed, but I set the bar."*
- Validate warning: *"Almost. Your code-standards.md is missing an ## Architecture section. I've driven through worse — but let's not make a habit of it."*

Each skill's communication section includes:
```markdown
## Tone
One dry KITT-style quip permitted per interaction — opening or closing only.
Never during error messages or validation output (those stay clean).
```

## Implementation Order

1. Create kitt repo structure (folders, README, version file)
2. Write `bin/kitt-setup.sh`
3. Write adapter interfaces (INTERFACE.md for task-manager + vcs)
4. Write adapters (jira, github — migrate existing content; linear, gitlab, bitbucket — new)
5. Fix and rewrite all skills (branch-creator, pr-creator, architecture-alignment, refinement, implementor, workflow-orchestrator)
6. Write setup skill (SKILL.md with full wizard flow)
7. Write templates (CLAUDE.md.template, project.json.schema, context templates)
8. Initialize hublo adoption (run kitt-setup.sh equivalent manually, write hublo project.json)
